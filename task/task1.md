# 任务背景
现在我们在参加一个比赛，目的是从各个公司财报里面提取出来所有有用的信息，我现在负责的是后端的部分。我先把比赛的内容贴给你

一、赛题：基于大模型的上市公司财报分析系统
1.	题目描述
随着大模型技术的快速发展，其在金融、商业分析等领域的应用潜力日益凸显。上市公司财报是投资者、监管机构和市场分析师了解企业经营状况的核心信息来源，但财报通常包含海量非结构化文本（如管理层讨论、业务展望）和半结构化数据（如财务报表、业务分类），人工提取关键信息的效率较低且易受主观因素影响。
本赛题聚焦于利用大模型技术自动化提取上市公司财报中的经营指标数据，旨在推动大模型在金融信息结构化、企业情报自动化等场景的落地应用。
基于预定义的公司列表、经营指标列表，依次从每家公司2024年发布的年报中提取经营指标值(公司列表、经营指标列表见附件，年报可以从巨潮资讯网下载：http://www.cninfo.com.cn/new/index)
2.	题目要求
(1)	提取指标的值和单位；
(2)	若财报中披露了去年同期值、前年同期值、年同比等衍生数值，需要一并提取；
(3)	使用多个大模型分别提取指标，融合多个模型提取的结果，对每个指标的提取结果进行可信度分类；
(4)	提取的每个指标值支持溯源，需要返回指标所在段落坐标。
3.	赛题评价
(1)	设计质量，系统架构与文档质量（20%）；
(2)	结果质量，模型提取指标值与人工标注的F1值（40%）；
(3)	体验性，所有财报处理的平均耗时（20%）；
(4)	创新性，模型对财报经营指标提取方法的创新性（20%）。
4.	限制条件
(1)	年报文档格式为PDF格式，需要对PDF格式的文档进行解析，准确地识别目录结构、段落文本和表格，解析后的格式能较好地还原PDF内容。
5.	附件（赛题输入）：
 
(1)	公司列表sheet包含目标公司名称和股票代码；
(2)	经营指标sheet 包含每家公司的经营指标内容；
(3)	采集样例sheet 包含指标采集示例。

# 前后端交互的规范：

结构规范说明
没问题。为你的这个场景（财报关键词提取）设计一套清晰、RESTful 风格的 API 接口规范是最佳实践。

这个规范将遵循一个**异步处理**的模式：

1. 前端：上传 PDF。
2. 后端：接收文件，**立即**返回一个“任务ID”，然后开始在后台处理（提取关键词）。
3. 前端：使用这个“任务ID”定期（轮询）查询后端，直到后端返回“处理完成”及关键词数据。
  

---

API 接口规范（建议用JS的Axios库）

1\. 上传 PDF 并触发分析

这个接口用于客户端上传 PDF 文件。后端收到后，会为这个文件创建一个唯一的标识符（reportId）并开始分析任务。

- Endpoint: POST /api/v1/reports
- Method: POST
- Content-Type: multipart/form-data (因为包含文件)
  
Request (请求体):
multipart/form-data 表单应包含:

字段名
类型
描述
是否必须
file
File
用户上传的 PDF 财报文件。
是
fileName
string
文件名，如果前端想指定的话。
是

Response (成功响应):

- Status Code: 201 Created
- Body (JSON):
  

{
  "success": true,
  "message": "文件上传成功，正在处理中...",
  "data": {
    "reportId": "uuid-a1b2c3d4-e5f6-7890",
    "fileName": "2024年年度财报.pdf",
    "status": "PROCESSING",
    "uploadedAt": "2025-11-15T12:05:00Z"
  }
}

关键点： 后端**立即**返回 reportId 和 PROCESSING 状态，而不是等待分析完成。这避免了请求超时。


---

2\. 获取分析结果（关键词）

前端使用上一步获取的 reportId，通过这个接口来查询分析的状态和结果。

- Endpoint: GET /api/v1/reports/{reportId}
- Method: GET
  
Request (请求):

- URL 路径参数: {reportId} (例如: /api/v1/reports/uuid-a1b2c3d4-e5f6-7890)
  
Response (响应):

- Status Code: 200 OK
  
A. 响应体 (仍在处理中):

如果后端还在分析，它会返回：

{
  "success": true,
  "data": {
    "reportId": "uuid-a1b2c3d4-e5f6-7890",
    "status": "PROCESSING",
    "keywords": [] // 关键词为空
  }
}

B. 响应体 (处理完成):

一旦后端分析完毕，它会返回：

{
  "success": true,
  "data": {
    "reportId": "uuid-a1b2c3d4-e5f6-7890",
    "status": "COMPLETED",
    "message": "解析PDF文档成功",
    "keywords": {
      "债券面值": {
        "value": "1500.00",
        "value_lastyear": "123456",
        "value_before2year": "123456",
        "YoY": "21.57%", //同比增长
        "YoY_D":"-10345", //同比增长额
        "unit": "百万元",
        "year": "2022", //预留提取年份，可能不一定会提取
        "type": "actual",
        "confidence": 100, //改为使用百分数
        "page_id": 1,
        "para_id": 2
        // "support": ["glm-4-plus", "spark-4.0Ultra"], //
        // "notes": ["..."] // 这两个字段可能想删掉
      }
    }
  }
}


C. 响应体 (处理失败):

如果后端分析失败（例如 PDF 已损坏或无法解析）：

{
  "success": true, // 注意：API请求本身是成功的
  "data": {
    "reportId": "uuid-a1b2c3d4-e5f6-7890",
    "status": "FAILED",
    "message": "无法解析此 PDF 文档。",
    "keywords": []
  }
}


---

完整的前后端交互流程

1. [前端] 用户选择一个 PDF 文件。
2. [前端] 调用 POST /api/v1/reports，将文件以 multipart/form-data 形式上传。
3. [后端] 接收文件，保存，生成 reportId，开始异步执行“关键词提取”任务。
4. [后端] **立即**返回 201 Created 和 JSON 体：{ "data": { "reportId": "uuid-...", "status": "PROCESSING" } }。
5. [前端] 收到响应，获得 reportId。此时，前端可以显示一个“处理中”的加载动画。
6. [前端] **开始轮询 (Polling)**：
  - 立即（或延迟 1-2 秒后）调用 GET /api/v1/reports/{reportId}。
  - 如果收到的 status 仍为 PROCESSING，则等待 3-5 秒后再次调用该接口。
7. [后端] “关键词提取”任务完成，将结果（keywords 数组）和 COMPLETED 状态存入数据库。
8. [前端] 下一次轮询 GET /api/v1/reports/{reportId} 时。
9. [后端] 查到状态已变为 COMPLETED，返回 JSON 体：{ "data": { "status": "COMPLETED", "keywords": [...] } }。
10. [前端] 收到 status: "COMPLETED" 和 keywords 数组。
11. [前端] 停止轮询。
12. [前端] 将 keywords 数组传递给你之前提到的 @react-pdf-viewer/search 插件的 search() 方法，实现高亮。
